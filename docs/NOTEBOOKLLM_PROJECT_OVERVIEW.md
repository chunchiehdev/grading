# AI-Powered Grading System: 完整專案彙整 (NotebookLLM Ready)

> **目的**: 供 NotebookLLM 快速理解專案全景，用於生成教育導向的影片介紹
> **受眾**: 教育工作者、管理者、AI 教育愛好者（非技術人員為主）
> **風格**: 故事導向 + 學習理論框架，避免過度技術細節

---

## 📋 Part 1: 問題背景與機會

### 1.1 核心痛點

**情景**: 一位高中國文老師 Ms. Lin

- 每週收到 **180 篇學生作文**（30 名學生 × 6 篇/週）
- 每篇評改需時 **8-15 分鐘**（含思考、記錄反饋）
- **每週工作量**: 24-45 小時
- **時間分配失衡**:
  - 💾 機械工作（逐行檢查、計分）: **70%** 的時間
  - 💡 高價值工作（指導學生、釐清概念）: **30%** 的時間
- **結果**:
  - 學生反饋延遲 **3-5 天**（或更久）
  - 教師身心俱疲，難以進行一對一教導
  - 課堂時間浪費在重複講解評分標準

### 1.2 教育洞察（Learning Insights）

根據教育研究：

- **Hattie 的 Visible Learning**: 反饋是影響學習效果最高的因素（effect size: 0.73）
- **但前提**: 反饋必須**及時**（ideally < 24h）且**具體**（linked to rubric）
- **目前困境**: 教師想給好反饋，卻被時間淹沒

### 1.3 解決方案的願景

> **使用 AI 做「初步篩選與標記」，讓教師專注「高階指導與人性化評論」**

```
舊流程: 學生提交 → (3天等待) → 教師逐篇手改 → (反饋籠統) → 學生收到不清楚的評語

新流程: 學生提交 → (15分鐘) → AI初評+標記 → 教師快速審閱+補充評論 → (當天) → 學生收到「AI邏輯+教師洞察」
```

## 🎯 Part 2: 系統核心功能架構

### 2.1 三大角色與流程

#### **教師 (Teacher) 旅程**

1. **課程設計** → 建立課程 > 邀請學生（邀請碼）
2. **評分標準定義** → 建立 Rubric
   - 例: 文章寫作評分
     - 結構清晰: 10 分 (必須包含開頭、中段、結尾)
     - 論點支撐: 10 分 (至少 3 個例證)
     - 語言流暢度: 5 分 (無句式錯誤)
     - 總分: 25 分
3. **作業發布** → 設置截止時間 > 上傳參考資料
4. **批改監督** → 看到 AI 草稿評分 → 點擊「檢視」→ 修改或確認 → 發送最終反饋

#### **學生 (Student) 旅程**

1. **加入課程** → 用邀請碼進入
2. **瀏覽作業** → 查看清單 > 點擊看詳細要求與 Rubric
3. **上傳作業** → PDF 檔案（系統自動解析文字）
4. **收到反饋** → 
   - **AI 部分**: "根據你的文章，結構評分得 8/10，因為 [具體原因]"
   - **教師部分**: "你的例子選得很好，但第二段論述還需加強"
5. **改進迴圈** → 下載教師回饋 > 修改作業 > 重新提交（如允許）

### 2.2 四層技術架構（簡化版）

#### **層 1: 前台 (User Facing)**
- 教師儀表板: 課程管理、Rubric 設計、提交列表、評分審核
- 學生平台: 課程清單、作業詳情、上傳頁、反饋查看

#### **層 2: 資料交換**
- API 層: 作業上傳、提交查詢、評分儲存、WebSocket 推送
- 檔案系統: 所有 PDF 儲存在 MinIO（私有 S3）

#### **層 3: 核心智能**
- **評分隊列** (BullMQ): 
  - 每個提交成為一個「任務」(Job)
  - 系統排隊處理（防止 API 超限）
  - 支援重試與失敗回退
- **AI 評分引擎**:
  - 讀取評分標準 (Rubric) 
  - 解析學生作業 (PDF → 文字)
  - 呼叫 Google Gemini (首選) 或 OpenAI (備用)
  - **核心 Prompt**: "根據這份 Rubric 評分這份作業，並解釋每一項的評分理由"
  - 回傳: 各項分數 + 詳細理由 + 改進建議

#### **層 4: 即時通知**
- WebSocket 伺服器: 當評分完成時，立即推送通知給教師與學生
- Redis Pub/Sub: 多個系統實例間的訊息同步

### 2.3 資料流範例

```
[Student 上傳] 
  ↓ 
[檔案進入 MinIO] 
  ↓ 
[建立 Job 進入 BullMQ 隊列]
  ↓
[背景 Worker 讀取: Rubric + PDF]
  ↓
[呼叫 AI API]
    ├─ 成功 → 儲存評分結果
    └─ 失敗 (rate limit) → 重試或降級
  ↓
[結果寫入資料庫]
  ↓
[WebSocket 推送通知]
  ↓
[教師看到「待審核」徽章] [學生看到「評分完成」通知]
```

## 🧠 Part 3: 應用的學習理論

### 3.1 形成性評量 (Formative Assessment)

**理論基礎**:
- 評量不只是期末考
- 持續的、低風險的反饋能引導學習

**系統實踐**:
- AI 評分後，學生看到 **即時反饋**（無評分壓力，純指導）
- 教師的最終評語加入 **個性化觀察**
- 支援 **重新提交**，形成循環

**影片表現**:
- 場景: 學生坐在電腦前，提交後 15 分鐘看到「你的結構很清晰，但論據需要更多」
- 對比: 舊時代是「過了一週老師才說 A-」

---

### 3.2 反饋品質與時機 (Feedback Quality & Timeliness)

**理論基礎** (Hattie, Dunning-Kruger):
- 有效反饋必須:
  - ✅ **及時** (within 24h 最佳)
  - ✅ **具體** (not "好" but "你的論點在第二段缺少支持")
  - ✅ **可操作** (告訴學生該怎麼做)
  - ✅ **透明** (為什麼這樣評分)

**系統實踐**:
- **及時**: BullMQ + Redis 架構 → 95% 評分在 15 分鐘內完成
- **具體**: Rubric 提供結構 → AI 按項目評分 → 連同理由回傳
  ```
  例: 而不是說 "不錯" 
      系統說 "結構評分 8/10 因為包含完整開頭中段結尾，但轉折銜接略嫌生硬"
  ```
- **可操作**: AI 附加建議，教師補充個案指導
- **透明**: 顯示評分依據、AI 信心度、教師審核標記

**影片表現**:
- 分割畫面對比: 左邊舊反饋（"C+，需加油"），右邊新反饋（具體項目分 + 原因）
- 強調: "從 2-3 天到 15 分鐘" 的時間差

---

### 3.3 標準化評分標準 (Rubric-Based Learning)

**理論基礎**:
- 明確標準幫助學生**自我評估**
- 減少評分的**主觀性**與**焦慮**
- 激發學生**自主改進**

**系統實踐**:
- 教師花時間設計 **Rubric** 而非逐篇評分
  - 一次性投入 (30 min Rubric) vs. 360 小時手評
  - AI 自動 apply Rubric
- 學生看到明確標準，知道如何自我調整
- 系統支援 Rubric 版本控制 → 可追蹤評分標準的演化

**影片表現**:
- 展示 Rubric 設計介面（教師 POV）
- 跳到學生 POV：看著 Rubric 重新檢查自己的作業
- 字幕: "好的標準 = 學生自主學習的指南"

---

### 3.4 認知負荷理論 (Cognitive Load Theory)

**理論基礎** (Sweller):
- 人腦同時處理能力有限
- 移除不必要的認知負荷，才能專注核心學習

**系統實踐**:
- **教師角度**: 減少 "計分" 和 "記錄" 的負荷
  - AI 打分 + 提示 → 教師只需「決定是否同意」+ 補充評論
  - 從 15 分鐘/篇 → 2-3 分鐘/篇
  - 釋放時間給 **高階教學**: 一對一討論、概念澄清、激發興趣
- **學生角度**: 清晰的 Rubric 和反饋結構
  - 不用猜老師在想什麼
  - 直接看到改進方向

**影片表現**:
- Ms. Lin 在下午 4 點下班（舊時代是 8 點才離開辦公室）
- 字幕: "AI 處理機械工作，教師回歸教學本質"

---

### 3.5 信任與透明度 (Trust & Transparency)

**理論基礎**:
- AI 評分容易招致懷疑 ("為什麼要相信機器？")
- 透明的推理 + 教師審核 = 信任建立

**系統實踐**:
- AI 評分時提供:
  - 各項理由 (不只是分數)
  - 信心度分數 (低於 60% 自動標記「需教師審查」)
  - 思考過程摘要
- 教師看到 AI 評分後可以:
  - 認同 ✓ (快速確認)
  - 修改 (改分數 + 原因)
  - 補充評論
  - 每個決定都留下記錄（可追蹤）
- 學生看到:
  - "AI 認為你的論點在…，分數 7/10"
  - "您的教師同意此評分並補充："

**影片表現**:
- 三角色對話:
  - AI: "根據 Rubric…我認為分數是…(我的信心度 85%)"
  - 教師: "我同意，並補充…"
  - 學生: "現在我明白為什麼了"
- 強調: "AI + Human Judgment = 最佳評分"

## 📊 Part 4: 業務成果與影響

### 4.1 時間節省

| 指標 | 舊流程 | 新流程 | 節省 |
|------|-------|-------|------|
| 每週作業評分總時數 | 40 小時 | 18 小時 | 55% |
| 單篇評分時間 | 12 分鐘 | 2-3 分鐘 | 75% |
| 反饋交付時間 | 3-5 天 | 15 分鐘 | 99% |
| 教師周末工作時間 | 8-10 小時 | 1-2 小時 | 80% |

### 4.2 教學品質提升

- ✅ 反饋更及時 → 學生改進更快
- ✅ 反饋更具體 → 學生更清楚如何改進
- ✅ 教師時間多 → 更多一對一指導、課堂創新設計
- ✅ 評分更一致 → Rubric 標準化

### 4.3 學生體驗改善

- 📱 即時通知: "你的作業評分完成了！"
- 📝 清晰反饋: 知道每一項如何評分及為什麼
- 🔄 改進循環: 可重新提交，看新的反饋
- 📚 自主學習: Rubric 清晰，學生可自我檢查

## 🔧 Part 5: 技術如何實現這一切（非技術要點）

### 5.1 三個關鍵決策

#### 1️⃣ **後台任務隊列 (Job Queue)**
- **為什麼**: AI API 有速度限制 (rate limit)
  - 同時 100 個學生提交 → API 會拒絕
  - 解決: 排隊等候，逐個處理
  - 效果: 99.7% 評分在 15 分鐘內完成
  - 用戶不會看到「系統太慢」

#### 2️⃣ **雙層評審系統**
- **AI 層**: 快速初評 + 自動標記「需人工審核」(低信心度)
- **人工層**: 教師決定是否修改 + 加入個性化評論
- 優點: 自動化節省時間 + 保留人類判斷

#### 3️⃣ **即時通知 (WebSocket)**
- **為什麼**: 沒人想一直按「重新整理」查看進度
- **怎樣**: 評分完成後，系統主動推送到瀏覽器
- 效果: 教師不用等待，專注其他工作

### 5.2 數據安全性（簡述）

- 所有學生檔案 **加密儲存** 在 MinIO
- 評分過程 **不留存** 原始作業到外部 API
- 教師與學生資料 **隔離**
- 所有操作 **有日誌追蹤**

## 附錄: 快速參考 (QA)

**Q: 這個系統是不是要取代教師？**
A: 不是。AI 做的是 30% 的機械工作（計分、記錄），讓教師把 70% 的時間投入高階教學。

**Q: 學生會信任 AI 評分嗎？**
A: 系統設計讓每個評分都可追蹤理由，且由教師最終審核。透明度 = 信任。

**Q: 如果 AI 評錯了怎麼辦？**
A: 教師一眼就能看出，點擊修改即可。所有改動都留下記錄。

**Q: 對小班級課程有幫助嗎？**
A: 有。即使只有 10 名學生，反饋品質提升 + 教師時間釋放仍然很有價值。

**Q: 支援多種作業形式嗎？**
A: 目前專注文字型（PDF 作業）。未來可擴展到程式、影片、創意作品等。
